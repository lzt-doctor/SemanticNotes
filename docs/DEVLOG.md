## Phase 7: Q&A(RAG)(2026-07-08)

### やったこと

- RAG パイプラインを「検索(既存の SearchIndexService を再利用)→ 根拠の組み立て(RAGPromptBuilder)→ 生成(AnswerGenerator)」の3層で実装。
- `AnswerGenerator` プロトコル+ `FoundationModelAnswerGenerator`(OS 同梱オンデバイス LLM、通信なし)+テスト用モック。利用可否は `SystemLanguageModel.default.availability` で判定し、理由別の案内文(非対応端末 / Apple Intelligence 無効 / モデル準備中)へ変換。
- コンテキスト長対策: Phase 1 のトークン見積もりを再利用し、根拠の合計を予算1,500トークン以内で上位優先に選抜。指示文で「渡した抜粋だけを根拠に、無ければ『見つかりませんでした』、出典番号 [n] を引用」と制約。
- `QAView` + `QAViewModel`(MVVM)。回答の下に根拠ノート(番号+タイトル+抜粋、タップで原文へ)を必ず表示。生成が使えない環境では検索結果のみ+案内を表示。
- `SearchIndexService` に `topChunks`(チャンク単位検索)を追加し、既存のノート単位検索をその上に再構成(重複排除)。
- テスト8件追加(計54件): プロンプト構築、予算選抜、フォールバック、モック生成、利用可否記録、対応環境のみの実生成テスト。

### 迷った判断

- **フォールバックの粒度**: 「Q&A 画面ごと無効化」でなく「生成だけ無効化して検索結果(根拠)は見せる」を選択。アプリの核は検索と根拠提示であり、生成はその上の薄い層という設計思想を UI にも反映した。
- **セッションを質問ごとに使い捨て**: 会話履歴を持たせると文脈窓を履歴が食いつぶす。履歴なしなら窓を根拠と質問だけに使え、コンテキスト管理が予算1つで済む。
- **出典と回答の整合**: 画面の出典リストは「実際にプロンプトへ入れたチャンク」だけに限定(予算で落ちたチャンクを出典として見せない)。
- **Chunker を nonisolated 化**: 既定 MainActor 分離のままだと nonisolated な PromptBuilder から呼べないというコンパイルエラーで発覚。純粋関数は明示的に nonisolated にする方針を確認した。

### 動作確認

- 全54テスト成功。シミュレータでは Foundation Models が `unavailable(appleIntelligenceNotEnabled)` を返し、**フォールバック経路が実環境で発動することを確認**(案内文も想定どおり表示)。
- 対応実機(Apple Intelligence 有効)での生成は `FoundationModelIntegrationTests` が自動で有効化される。実機確認は次回の実機接続時に行う。

### CI 障害と対処(2026-07-08 追記)

## Phase 6: 評価(2026-07-08)

### やったこと

- 自作の日英ミニベンチマークを作成(`SemanticNotesTests/Resources/BenchmarkDataset.json`)。ノート200件(20トピック×10件、日本語系120・英語80)+クエリ40件。クエリは本文と語彙が重ならない意味的な言い換えで、等級付き正解ラベル(2=直接、1=関連)を付与。
- `quantize_model.py` を追加し、INT8(weight-only 線形量子化)版モデルを生成。install_model.sh が両方を配置するよう更新。
- `EvaluationBenchmarks`(環境変数で有効化)で recall@10 / nDCG@10 / レイテンシを FP16・INT8 × 総当たり・HNSW の4構成で測定。言語別内訳と取りこぼしクエリの一覧も出力。
- 結果を `docs/RESULTS.md` に表と図(自作 SVG 3枚: レイテンシ×N、efSearch×recall、サイズ×品質)でまとめた。

### 迷った判断

- **正解ラベルの設計**: 総当たり結果を正解にする案(ラベル不要)は「モデル自体の質」が測れないため、人手ラベルを採用。等級は2段階に抑え、ラベルの主観性は RESULTS の限界節に明記した。
- **nDCG の利得**: 2^等級−1(等級2=3.0、等級1=1.0)。直接の答えを関連の3倍重く扱う標準的な定義。
- **量子化方式**: activation 込みの INT8 は変換の難度と精度リスクが上がる割に、サイズはほぼ重みで決まる。weight-only を採用し「劣化源を重みの丸め誤差だけに限定する」設計にした。
- **評価テストのバグ**: 配列リテラルが `Bundle.url(forResource:)` の `String?` 引数に引きずられて `[String?]` に推論されるという Swift の型推論の罠を踏んだ(型注釈で解決)。

### 計測結果(詳細は docs/RESULTS.md)

- 検索品質(FP16・総当たり): **recall@10 = 0.850、nDCG@10 = 0.690**(JA 0.875/0.706、EN 0.813/0.666)。
- HNSW の近似劣化: **ゼロ**(真 top-10 一致率 1.00、ラベル指標も総当たりと同一)。
- INT8 量子化: **サイズ 235.3 → 118.5 MB(50%)**、nDCG@10 は 0.690 → 0.686(−0.5%)、recall@10 不変。PyTorch との最小 cos は 0.999832。
- 取りこぼし9クエリの大半は等級1ラベルが圏外だったもので、等級2はほぼ全クエリで上位に入った。
- 埋め込み: passage 約42ms/件、クエリ約30ms/件(シミュレータ)。

## Phase 5: HNSW 自作(2026-07-08)

### やったこと

- `HNSWIndex`(actor)を論文(Malkov & Yashunin 2016)に沿って自作。層の確率的割り当て(1/M)、上層の貪欲降下、第0層のビーム探索(Algorithm 2)、多様性ヒューリスティックによる近傍選択(Algorithm 4)、swap しない墓標方式の削除+しきい値超過での再構築。
- 補助として `BinaryHeap`(約50行)も自作し、単体テストで検証。
- ディスク永続化: 独自バイナリ形式(magic/版数/次元の検証、リンク先の範囲チェック付き)。保存時に墓標を再構築で回収してから書き出す。
- `VectorIndex` プロトコルへの差し替えテスト(HNSW + モック埋め込みで検索パイプラインがそのまま動く)。アプリの既定は Phase 6 の評価まで BruteForce のまま。
- テスト13件追加(計44件)+ 環境変数 `TEST_RUNNER_RUN_BENCHMARKS=1` で走る重い計測3件。

### 迷った判断・学び

- **削除**: リンク繋ぎ直しは品質を壊しやすいため、墓標(通過は許すが結果に出さない)+「墓標 > 64 かつ生存数の半分」で全再構築、という2段構え。
- **ベンチマークのデータ分布が結果を支配した**: 最初は一様ランダムな 384 次元ベクトルで計測し、recall@10 が N=10,000 で 0.46 まで落ちた。原因は実装バグではなく「次元の呪い」(一様高次元データは全ペアがほぼ等距離で、グラフ探索の手がかりが消える)。efSearch を上げると recall が単調に 1.0 へ回復すること(64→0.67、128→0.86、256→0.97、512→1.0)を確認し、グラフ自体は健全と判定。実際の文埋め込みは話題ごとのクラスタ構造を持つため、代表ケースをクラスタ混合データ(中心+ガウスノイズ)に変更した。実データでの最終検証は Phase 6 で行う。
- **計測コードのバグを1つ踏んだ**: HNSW と総当たりに別々の UUID を振っていて recall が常に 0 になった。recall は ID 集合の比較なので同じ ID を共有する必要がある。「recall がきっかり 0」は実装より計測系のバグを疑うべき、という教訓。

### 計測結果(iPhone 17 シミュレータ / Debug ビルド、クエリ50件平均、シード固定)

代表ケース(クラスタ構造データ、M=16, efC=200, efS=64):

| N | 構築時間 | recall@10 | HNSW 検索 | 総当たり検索 | 倍率 | グラフ追加メモリ |
|---|---|---|---|---|---|---|
| 1,000 | 2.0 s | 1.00 | 0.36 ms | 1.24 ms | ×3.5 | 0.15 MB |
| 5,000 | 13.8 s | 1.00 | 0.42 ms | 4.67 ms | ×11.2 | 0.74 MB |
| 10,000 | 31.8 s | 1.00 | **0.52 ms** | 9.25 ms | **×17.9** | 1.48 MB |

- **完了条件を達成**: recall@10 = 1.00(≥ 0.95)のまま、N=10,000 で総当たりの約18倍高速。倍率が N とともに拡大 = O(log N) vs O(N) の理論どおり。
- グラフの追加メモリはベクトル本体(N=10,000 で約15MB)の1割以下。
- 構築は1点ずつの逐次挿入で N=10,000 に約32秒(Debug)。永続化で再構築を回避できる。

パラメータ影響(クラスタ構造データ、N=5,000):

| M | efC | 構築 | recall@10 | 検索 |
|---|---|---|---|---|
| 8 | 100 | 4.8 s | 1.000 | 0.35 ms |
| 8 | 200 | 7.9 s | 0.998 | 0.37 ms |
| 16 | 100 | 9.7 s | 1.000 | 0.39 ms |
| 16 | 200 | 14.2 s | 1.000 | 0.45 ms |
| 32 | 100 | 23.7 s | 1.000 | 0.47 ms |
| 32 | 200 | 29.3 s | 1.000 | 0.46 ms |

efSearch 掃引(M=16, efC=200): efS=10 → recall 0.986 / 0.12ms、16 → 0.996 / 0.16ms、32 → 1.0 / 0.29ms、64 → 1.0 / 0.41ms、128 → 1.0 / 0.81ms。**recall は早期に飽和し、時間は efS にほぼ比例** — 事前に説明したトレードオフ曲線そのもの。

- クラスタ構造データでは M=8 でも recall がほぼ 1.0 → 既定値(M=16, efC=200, efS=64)は安全側。実データが難しければ efSearch を上げる余地が大きいことも一様ランダム実験で確認済み。

## Phase 4: 総当たり検索と検索 UI(2026-07-08)

### やったこと

- `VectorIndex` プロトコルと `BruteForceIndex`(actor)を実装。全ベクトルを row-major の連続配列に平坦化し、`vDSP_mmul`(N×d 行列 × クエリ)**1回**で全チャンクとの内積を計算。削除は swap-remove で穴を作らない。
- `NoteChunk` に `chunkID: UUID` を追加し、インデックスのキーを SwiftData の内部 ID から分離(Phase 5 のディスク永続化とテスト容易性のため)。
- `SearchIndexService`: 検索画面を開くたびにストアとインデックスを全量同期(高コストな埋め込み計算は未計算チャンクのみ)。検索はチャンク単位の上位候補をノート単位に畳み、最良チャンクのスコアと抜粋を返す。
- `SearchView` + `SearchViewModel`(MVVM)。モデル未配置環境では検索だけを無効化して案内を表示。
- テスト10件追加(計31件): 索引の正確性4、レイテンシ計測1、サービス(モック)4、**実モデルでの意味検索統合テスト1**。

### 迷った判断

- **インデックスのキー**: SwiftData の `PersistentIdentifier` は Codable で扱いも楽だが、インデックスが SwiftData に結合する。`UUID` をチャンクに持たせて分離した(HNSW の永続化フォーマットにもそのまま使える)。
- **内積の計算**: `vDSP_dotpr` を N 回呼ぶより、平坦化した行列への `vDSP_mmul` 1回の方が呼び出しオーバーヘッドとキャッシュ効率で有利。
- **インデックス更新**: 差分反映は追従漏れ(削除ノートが検索に残る等)のリスクがあるため、再構築が安価な総当たりのうちは全量同期。再構築が高価になる HNSW(Phase 5)で差分管理を検討する。
- **上位 k の選択**: N が数万規模までは全ソートで十分(計測で確認)。部分選択アルゴリズムは必要になってから。

### 計測結果(iPhone 17 シミュレータ / M系 Mac、単独実行・ウォームアップ後20回平均)

| N(チャンク数) | 検索レイテンシ(k=10, d=384) |
|---|---|
| 1,000 | **1.28 ms** |
| 5,000 | **4.93 ms** |
| 10,000 | **9.41 ms** |

- レイテンシは N にほぼ線形 → **O(N·d) の理論どおり**。1チャンクあたり約 0.94µs。
- 検索全体ではクエリの埋め込み(約30ms)が支配的で、1万チャンクでも合計 約40ms。
- 実モデル統合テスト: 本文と語彙が重ならない意味的クエリ(「非同期プログラミングのメモ」→ actor のノート、「料理の手順」→ カレーのノート)で正しいノートが1位。

## Phase 3: EmbeddingService(2026-07-08)

### やったこと

### 迷った判断

- **swift-transformers のバージョン**: 最初 0.1.24 で入れたところ、`XLMRobertaTokenizer` がトークナイザレジストリ未登録で `unsupportedTokenizer` エラー。tokenizer_config.json の class 名を偽装する回避策(同じ Unigram 系の T5Tokenizer に振る)も検討したが、**1.3.3 で正式対応済み**と確認できたため、依存のバージョンを上げる正道を選んだ。1.x では推移的依存(swift-nio 等、Hub のダウンロード機能向け)が増えるが、アプリは Tokenizers 部分しか使わない。
- **トークナイザ不一致の検証方法**: コサイン類似度でなく**トークン ID の完全一致**で判定。不一致は「わずかに違うベクトル」という静かな壊れ方をするため、入口で厳密に検証する。
- **自動生成モデルクラスを使わない**: Xcode の codegen クラスを参照するとモデル未配置の CI でコンパイルが壊れるため、`MLModel(contentsOf:)` + バンドル検索で実行時解決にした。

### 計測結果(iPhone 17 シミュレータ / M系 Mac、ウォームアップ後5回平均)

- トークナイザ: Python(HF fast tokenizer)と**全8ケースでトークン ID 完全一致**。
- 埋め込み精度: PyTorch FP32 参照とコサイン **> 0.999**(全8ケース)、ノルム 1.000±0.01。
- 埋め込み時間: **クエリ(13トークン)約 30ms/回、チャンク相当の長文(165トークン)約 87ms/回**。可変長入力の効果でクエリは長文の約1/3の時間。実機での計測は Phase 4/6 で行う。

## Phase 1: チャンク分割(2026-07-07)

### やったこと

- `Core/Chunker.swift` を実装。段落(行)→ 文(NLTokenizer)→ 強制分割、という3段フォールバックで「必ず上限以下の単位」を作り、目標トークン数まで詰め込む。チャンク境界では直前チャンク末尾の単位をオーバーラップとして引き継ぐ(既定: 目標300 / 上限400 / オーバーラップ50)。
- トークン数は文字種ヒューリスティックで見積もる: CJK文字 ≈ 1文字1トークン、それ以外 ≈ 4文字1トークン(切り上げ)。実トークナイザ導入(Phase 3)までのつなぎで、多め(保守側)に見積もってモデル入力上限512を超えない方向に倒した。
- `NoteRepository.reindexIfNeeded(_:)` を追加し、create / update の保存フローで `needsReindexing` を消費してチャンクを全再生成。タイトルのみの変更では再生成しない。
- テストを16件に拡充(Chunker 単体9件+モデル/リポジトリ統合7件)。日英混在・空入力・句読点なし長文・オーバーラップ検証・孤児チャンクチェックを含む。サンプル文はすべて自作。

### 迷った判断

- **トークン見積もり**: 実トークナイザ待ち vs 文字種ヒューリスティック。後者を採用し、Phase 3 で実測トークン数と突き合わせて見積もり係数を検証する予定。
- **オーバーラップの単位**: 文字数で機械的に重ねる案もあったが、段落・文という意味の通る単位ごとに重ねる方式にした。境界をまたぐ情報の取りこぼし防止という目的に合い、テストでも検証しやすい。
- **フラグの消費タイミング**: チャンク分割は軽い文字列処理なので保存時に同期実行。重い埋め込み計算は Phase 3 以降で `embedding == nil` のチャンクだけを非同期処理する設計にする(フラグは異常終了時の取りこぼし防止も兼ねる)。
- **全再構築 vs 差分更新**: 正しさが自明な全再構築を採用。差分化は埋め込みコストが現実になる Phase 4 で計測してから判断。
- **文分割の実装**: 正規表現でなく OS 標準の NLTokenizer を採用。テスト作成中に「句点が2つある文字列は2文に割れる」という正しい挙動を確認し、テスト側の前提を修正した(実装は変更なし)。
