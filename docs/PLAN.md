## Phase 0: 環境確認と CI

- 目的: 開発の土台を固める。
- 作業: ビルドとテスト(既存4件)が通ることを確認。.gitignore を整備(ビルド生成物、モデル成果物、Python venv)。GitHub Actions でビルド+テストの CI を設定。docs/DEVLOG.md を作成。
- 完了条件: ローカルと CI の両方でテストが緑になる。
- 説明できるべきこと: CI を最初に入れる意義。iOS プロジェクトの CI が何をしているか。

## Phase 1: チャンク分割

- 目的: 検索の最小単位を作る。
- 作業: Chunker を実装(段落ベース+前後オーバーラップ、1チャンクの目安は200〜400トークン相当)。ノート保存時に NoteChunk を生成・更新する処理(needsReindexing フラグを活用)。日英混在テキストでの単体テスト。
- 完了条件: 分割の単体テストが通り、ノート編集後にチャンクが正しく更新される。
- 説明できるべきこと: なぜノート全体でなくチャンク単位で検索するのか。オーバーラップの役割。

## Phase 2: モデル変換パイプライン(Python)

- 目的: multilingual-e5-small をアプリで使える形にする。
- 作業: scripts/ に venv 手順と requirements.txt(バージョン固定)。convert_model.py で平均プーリング+L2正規化を含めた形で FP16 の .mlpackage を出力。validate_model.py で PyTorch と Core ML の出力ベクトルのコサイン類似度が 0.999 超であることを確認。Phase 3 のテストで使う「入力文と参照ベクトル」のペアを JSON で書き出す(日英両方、"query: " / "passage: " 両接頭辞を含める)。
- 完了条件: 変換と検証が私の Mac で再現できる(手順を scripts/README.md に明記)。
- 説明できるべきこと: プーリングと正規化をモデル側に焼き込む理由。FP16 化で何が変わるか。

## Phase 3: EmbeddingService(Swift)

- 目的: アプリ内で文をベクトルにできるようにする。
- 作業: swift-transformers でトークナイズし、Core ML で推論する EmbeddingService を実装。プロトコル+モックを用意。Phase 2 の参照ベクトルと一致することを確認するテスト(コサイン類似度で判定)。1チャンクあたりの埋め込み時間を計測。
- 完了条件: 一致テストが通る。計測値を DEVLOG に記録。
- 説明できるべきこと: トークナイザの不一致が起きると何がどう壊れるか。

## Phase 4: 総当たり検索と検索 UI

- 目的: 意味検索を最初に動く形にする。
- 作業: VectorIndex プロトコルと BruteForceIndex(vDSP の内積)を実装。検索画面(クエリに "query: " を付与→埋め込み→上位k件をノート単位で表示)。インデックス構築の非同期処理。N = 1,000 / 5,000 / 10,000 チャンクでの検索レイテンシ計測。
- 完了条件: 実データで意味検索が動き、計測値が DEVLOG にある。
- 説明できるべきこと: 「正規化+内積=コサイン類似度」の等価性。総当たりの計算量 O(N·d)。

## Phase 5: HNSW 自作

- 目的: 近似最近傍探索を自分の手で実装する(このプロジェクトの技術的な山場)。
- 作業: 実装前に HNSW のアルゴリズムを図と文章で私に説明し、理解を確認してから着手すること。HNSWIndex の実装、ディスク永続化、パラメータ(M, efConstruction, efSearch)の影響調査。総当たりとの recall@10・速度・メモリの比較ベンチマーク。
- 完了条件: recall@10 ≥ 0.95 を保ちつつ総当たりより大幅に高速(数値を DEVLOG へ)。
- 説明できるべきこと: HNSW が速い直感的な理由。recall と速度のトレードオフ。

## Phase 6: 評価

- 目的: 出願レポートの核になる定量結果を作る。
- 作業: 自作の日英ミニベンチマーク(サンプルノート数百件+クエリ30〜50件。データはすべて自作)。recall@10 / nDCG@10 / レイテンシ / メモリを測定。FP16 と量子化版(INT8 等)の品質・サイズ比較。結果を docs/RESULTS.md に表とグラフでまとめる。(発展、任意)別の埋め込みモデルとの比較。
- 完了条件: RESULTS.md が出願レポートにそのまま転用できる水準になっている。
- 説明できるべきこと: 各指標の意味。量子化が品質に与えた影響とその理由。

## Phase 7: Q&A(RAG)

- 目的: 検索結果を根拠にした回答生成まで完成させる。
- 作業: RAGService(Foundation Models)を実装。利用可否チェックと非対応環境のフォールバック。検索上位3〜5チャンクをコンテキストに回答を生成し、出典(どのノートか)を表示。
- 完了条件: 対応実機で動作し、非対応環境ではフォールバックが動く。
- 説明できるべきこと: RAG の構成。オンデバイス LLM のコンテキスト長制約への対処。

## Phase 8: 仕上げ

- 目的: 初見の人に伝わる形に整える。
- 作業: UI の磨き込み。README(構成図・結果表・セットアップ手順。出願先が英語圏なら英語版も)。デモ動画の撮影台本。TestFlight 配布の準備。
- 完了条件: README だけを読んだ人がプロジェクトの内容と成果を理解できる。
- 説明できるべきこと: このプロジェクトの貢献を1分で説明する台本。
